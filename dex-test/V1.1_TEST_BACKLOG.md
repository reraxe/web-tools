# Dex v1.1-test Backlog

This file tracks issues discovered while testing `v1.0-stable`. Changes are implemented and verified in `v1.1-test` without modifying the stable deployment or its SQLite data.

## Release scope

`v1.1-test` is the first consolidated quality-of-life release. It includes the work previously considered for `v1.1-test` through `v1.4-test`: essential fixes, multi-card batch organization, faster intake, safer daily controls, and complete exports.

The release must make manual inventory organization efficient and dependable. Automatic card recognition, synchronized card catalogs, marketplace intelligence, and additional game adapters begin with `v2.0-test`.

## 1. Camera QR scanning requires HTTPS

**Observed:** The phone displays: "Camera QR scanning needs HTTPS on this device. You can type the SKU for now."

**Cause:** Mobile browsers block camera access when Dex is opened over unsecured HTTP.

**Target:**

- Publish Dex through the intranet reverse proxy at `https://dex.home`.
- Use a certificate trusted by the phone.
- Keep manual SKU entry available as a fallback.
- Improve the UI message so it clearly distinguishes missing HTTPS from unsupported QR detection.

**Acceptance:** The phone can open the rear camera, scan a Dex QR label, and add the SKU to an outbound order.

## 2. Text must fit the 2 x 1 thermal label

**Observed:** The SKU and description area is cramped and can extend beyond its intended label field.

**Target:**

- Keep the QR code on the left and SKU/details on the right.
- Dynamically size or wrap long SKU text inside the available width.
- Keep the game and set description within the label boundary.
- Preserve exact 2 x 1 inch print dimensions without clipping.

**Acceptance:** The longest supported SKU and its description print completely inside a 2 x 1 label and remain readable on the iDPRT SP310.

## 3. Batch and SKU dates must use the business timezone

**Observed:** A batch created during the evening of June 20 received a June 21 date because the server uses UTC.

**Target:**

- Add a configurable timezone with `America/New_York` as the deployment setting.
- Use that timezone for batch codes, SKUs, sale dates, and displayed timestamps.

**Acceptance:** A batch created before local midnight uses the correct local calendar date regardless of the server's system timezone.

## 4. Bulk scan intake and image pairing

**Observed:** Browser intake requires adding one front/back pair at a time.

**Target:**

- Accept an entire batch of front/back images in one drag-and-drop action.
- Pair files automatically using front/back filenames when available, with sequential pairing as a fallback.
- Flag incomplete or ambiguous filename/order pairs in the review grid and provide a one-click front/back swap control.
- Assign a unique SKU to every physical card immediately.
- Apply the batch's game, set, color, condition, finish group, and location automatically.
- Show batch progress and allow pair correction before completion.
- Keep manual identification available for each card in the batch grid.

**Acceptance:** A complete scan batch can be dropped into Dex once, paired into physical-card entries, assigned unique SKUs, and efficiently identified from the batch grid.

## 5. Add cards to a completed batch

**Observed:** After a batch is finished, its intake form is hidden and there is no way to add cards that were missed or found later.

**Target:**

- Show an **Add more cards** button in the top-right batch status area when a batch is complete.
- Reopen the same batch and restore browser upload and watched-folder intake.
- Preserve every existing card, image, SKU, cost allocation, and printed-label status.
- Continue the SKU sequence for newly added physical cards.
- When the batch is finished again, queue labels only for new or otherwise unprinted cards.
- Record when the batch was reopened and completed again for audit history.

**Acceptance:** A completed batch can be reopened, receive additional cards, and be completed again without changing existing SKUs or reprinting old labels.

## 6. Completed batches are incorrectly labeled Sold

**Observed:** The inbound batch list displays a completed batch as **Sold**, while opening the batch correctly shows its card as **In stock**.

**Cause:** The batch list reuses card-status badges and incorrectly maps every non-open batch to `SOLD`.

**Target:**

- Use distinct batch statuses: **Open**, **Complete**, and later **Reopened**.
- Reserve **Sold** exclusively for individual cards processed through outbound.
- Optionally summarize available and sold card counts separately within each batch row.

**Acceptance:** Finishing intake changes the batch badge to **Complete** without changing or misrepresenting the inventory status of any card.

## 7. Batch workspace must support and display many cards

**Observed:** Although `v1.0-stable` can accept multiple cards before completion, intake is presented as a one-card-at-a-time form and completed batches do not provide a useful visual overview of all scanned cards.

**Target:**

- Treat every new batch as a multi-card intake workspace.
- Add all browser-uploaded or watched-folder scan pairs as separate physical-card entries in that batch.
- Display batch contents as a responsive grid with the card image, name, card number, SKU, rarity, variant, and review status.
- Keep duplicate physical copies as separate entries with their own SKUs.
- Allow sorting and filtering by scan order, card number, rarity, and needs-review status.
- Open an individual card from the grid to correct identification, swap front/back, or edit inventory details.
- Show batch totals for scanned, identified, needs review, and labels waiting.
- Preserve the grid after the batch is completed or reopened.

**Acceptance:** Opening a batch containing multiple scans shows every physical card in a clear grid, and each card can be reviewed or edited without leaving the batch workflow.

## Deferred to v2.0-test: Scheduled local card-catalog updates

**Observed:** A static reference library will not recognize newly released cards or corrected catalog records without being refreshed.

**Target:**

- Run configurable incremental catalog synchronization on a schedule.
- Fetch only new or changed metadata and reference images after the initial import.
- Show last successful sync, catalog card count, new records, errors, and next scheduled run.
- Provide a manual **Sync now** command.
- Keep the previous working catalog available if an update fails.
- Continue accepting scans while offline and retry unmatched cards after a successful sync.
- Store user-confirmed matches and corrections locally as additional reference evidence without uploading private scans.
- Respect source API rate limits, caching rules, and image-use terms.

**Acceptance:** Newly released cards become recognizable after the next successful catalog sync, and a failed update never interrupts normal inventory use.

## Deferred to v2.0-test: Automated marketplace pricing

**Observed:** Market low, average, high, and listing price must currently be entered manually for each card.

**Target:**

- Integrate authorized eBay and TCGplayer APIs or approved data providers; do not scrape marketplace pages.
- Refresh prices three configurable times per day in the business timezone.
- Provide **Refresh now** for an individual card, a batch, or all in-stock inventory.
- Match exact game, card number, printing, variant, finish, language, raw condition, and Near Mint status.
- Prefer recent completed sales where authorized data access is available and include buyer-paid shipping in delivered price.
- Exclude graded cards, lots, mismatched variants, damaged cards, and statistical outliers.
- Calculate low, average, and high from comparable sales and store source, sample count, confidence, and refresh timestamp.
- Detect price changes after each refresh and flag changes above a configurable percentage or dollar threshold.
- Keep suggested listing price separate from marketplace observations and require approval before altering an active listing.
- Cache results, respect rate limits, retry safely, and retain the previous valid price when a provider is unavailable.
- Store marketplace credentials as server secrets and never in GitHub or SQLite exports.

**Dependency:** Exact completed-sale coverage depends on the marketplace permissions and credentials available to the Dex deployment. Official API capabilities must be verified during implementation.

**Acceptance:** Recognized cards receive sourced low/average/high values automatically three times daily, can be refreshed on demand, and clearly show stale, low-confidence, or materially changed pricing.

## 10. Dashboard inventory counts must not overlap confusingly

**Observed:** The dashboard reports three items as **In Stock** while also reporting the same three items as **Needs Review**. Sold filtering itself works correctly.

**Cause:** The stable dashboard includes both `IN_STOCK` and `REVIEW` records in its In Stock total.

**Target:**

- Count only ready `IN_STOCK` cards under **In Stock**.
- Show `REVIEW` cards only under **Needs Review**.
- Optionally add a separate **Physically available** total when a combined count is useful.
- Ensure sold, held, review, and ready inventory metrics are clearly defined and non-overlapping.

**Acceptance:** Dashboard totals can be added together without double-counting the same physical card, and each status matches the inventory filter results.

## 11. CSV exports need complete sale financials

**Observed:** Inventory CSV correctly exports all five physical SKUs and their statuses, but sold records include only sold date, platform, and order number. Per-card sale price, shipping, fees, postage, and net proceeds are missing.

**Target:**

- Add per-card sale price and acquisition-cost allocation to the inventory export.
- Add a dedicated sales CSV with one row per order containing subtotal, shipping collected, platform fees, postage, and net proceeds.
- Include order identifiers and SKU line items without double-counting order-level amounts in totals.
- Preserve identifiers such as SKUs and order numbers as text.
- Use the configured business timezone in exported dates and filenames.

**Acceptance:** Inventory and sales exports can reproduce the Sales view, trace every sold SKU to its order, and calculate net proceeds without consulting the Dex database.

## 12. TCGplayer listing capacity must be configurable

**Observed:** `v1.0-stable` hardcodes the Level 1 capacity at 100 physical cards. The business is now a Level 2 seller with capacity for 500.

**Target:**

- Replace the hardcoded capacity with an editable seller setting.
- Set the current deployment limit to 500.
- Continue counting each physical listed copy against capacity.
- Show slots used, slots available, percentage used, and warnings near the limit.
- Allow future seller-level changes without rebuilding Dex.

**Acceptance:** The dashboard displays capacity as `used / 500`, and changing the configured limit updates all capacity indicators immediately.

## 13. Priority quality-of-life controls

**Target:**

- Add **Save and next** so successful intake clears the completed image pair, retains batch defaults, and immediately prepares the next card.
- Retain the current batch's rarity, variant, color, condition, and location until explicitly changed.
- Add **Undo last action** for accidental outbound sales, status changes, and batch completion, with a visible description of what will be restored.
- Add **Reprint label** for one physical SKU without marking unrelated labels printed or reprinting the entire batch.
- Add clear visual and optional sound feedback for successful scans, duplicate scans, unresolved pairs, and failed imports.
- Keep a short recent-action history so undo behavior is understandable and auditable.

**Acceptance:** Repetitive intake can proceed card after card without re-entering defaults, individual labels can be reprinted, and the most recent accidental inventory action can be safely reversed.

**Deferred candidates:** Persistent filters, quick drawer moves, richer recent-activity views, duplicate-copy shortcuts, and additional batch warnings remain candidates for a later test release.

## v2.0-test recognition milestone

- Build the Dex recognition pipeline for image cleanup, rotation, card-side detection, OCR, catalog matching, image similarity, and confidence scoring.
- Use OPTCG API (`https://optcgapi.com/`) as the primary One Piece catalog provider.
- Cache permitted One Piece metadata, reference images, and fingerprints locally so recognition remains fast and continues during provider outages.
- Route uncertain or conflicting matches to the batch review queue rather than silently guessing.
- Add scheduled and on-demand catalog synchronization after the local recognition workflow is reliable.
- Add marketplace pricing after exact card, printing, variant, finish, language, and condition matching can be trusted.

## Deferred: Pokemon catalog adapter

This is recorded for Version 2 so `v1.1-test` remains focused on efficient manual inventory organization.

- Use Pokemon TCG API as the preferred Pokemon catalog candidate.
- Use TCGdex as a fallback and cross-check where coverage permits.
- Test provider access directly from the Dex container when documentation is inaccessible from the development environment.
- Support versioned JSON/dataset imports when live API access is unavailable or unsuitable.
- Cache normalized metadata, permitted reference images, and fingerprints locally in SQLite/server storage.
- Continue recognition from the last successful local catalog during provider outages.
- Route missing or conflicting matches to manual review.
- Verify current licensing, attribution, caching, authentication, and rate-limit requirements before implementation.
